#### C1 — Module-level `stagesArray` mutated as shared mutable state

**Files:**
- `src/hooks/useProgressContext.tsx:56–119`
- `src/subjects/theory/curriculum/stages/helpers.ts:30`

**What's wrong:** `stagesArray` is a module-level singleton created once at import time. Five functions in `useProgressContext.tsx` mutate it in place:

```ts
// useProgressContext.tsx:56–68 — mutates the singleton directly
const refreshStageUnlockStatus = (...) => {
  stagesArray.forEach(stage => {
    stage.isUnlocked = calculateStageUnlockStatus(...)  // ← mutation
    stage.lessons.forEach(lesson => {
      lesson.isLocked = true  // ← mutation
    })
  })
}
```

This bypasses React's reconciler entirely. Components reading `stagesArray` after a mutation have no guarantee they're seeing fresh data — React doesn't know a change happened. The current workaround (a `refreshKey` state increment in both `TheoryScreenBody` and `AuralScreenBody` via `useFocusEffect`) is fragile and will break if a component reads stage data without the key trick.

**Fix:** Separate immutable lesson config from mutable progress state. Keep `stagesArray` as a read-only config source. Derive display state by combining config with `progressData` React state in a selector:

```ts
// Pure selector — no mutation, no module state
const getStageWithProgress = (
  stage: Stage,
  progressData: Record<string, ProgressData>
): Stage => ({
  ...stage,
  isUnlocked: calculateStageUnlockStatus(stage.id, stagesArray),
  lessons: stage.lessons.map(lesson => ({
    ...lesson,
    isLocked: progressData[lesson.id]?.isLocked ?? lesson.isLocked ?? false,
    stars: progressData[lesson.id]?.stars,
    isPassed: progressData[lesson.id]?.isPassed
  }))
})

// In consumers — derive from progressData (which IS React state)
const stages = useMemo(
  () => stagesArray.map(s => getStageWithProgress(s, progressData)),
  [progressData]
)
```

#### H1 — `ProgressContext.Provider` value object not memoized

**File:** `src/hooks/useProgressContext.tsx:449–474`

**What's wrong:** The context value object is created inline on every render:

```tsx
<ProgressContext.Provider

const contextValue = useMemo(() => ({

  progressData, revisionQuestions, revisionQuestionsLoading,
  progressDataLoading, progressDataInitialized,
  refreshProgress, refreshRevisionQuestions,
  updateLessonProgress, updateFinalTestProgress
])

return <ProgressContext.Provider value={contextValue}>{children}</ProgressContext.Provider>
```

#### H3 — `fetchUserData` not wrapped in `useCallback`

**File:** `src/hooks/useUserContext.tsx:41`

**What's wrong:** `fetchUserData` is defined as a plain arrow function, not `useCallback`. Every render creates a new function reference. Any consumer with `fetchUserData` in a dependency array will re-run unnecessarily (e.g. `HomeScreen` calls `refreshProfile` from it).

**Fix:**
```tsx
const fetchUserData = useCallback(async (): Promise<UserData | null> => {
  try {
    const result = await getUserData()
    // ... rest of body unchanged
  } catch (error) {
    // ...
  }
}, []) // no deps — only calls stable imported functions
```

Common pattern needing theme values passed as styled props:

```tsx
// BEFORE (index.tsx)
const theme = useTheme()
const cardSize = theme.dimensions.cardButtonSize

<Card size={cardSize} />

// AFTER — pass as styled prop, or accept as regular prop
// In ComponentName.styles.tsx:
export const ThemedCard = styled(Card)(({ theme }) => ({
  // use theme directly
}))
```

For cases where a theme value must be a prop (e.g. a library component like `Ionicons` that takes `size` as a number), extract the value into a styled wrapper or pass it via a typed prop from the parent:

```tsx
// LessonCard/LessonCard.styles.tsx
export const getCardSize = (theme: Theme) => theme.dimensions.cardButtonSize
// Then use it in a styled component callback — no useTheme() in index.tsx
```

#### H5 — React Compiler not configured (top Expo SDK 54 recommendation)

**File:** `babel.config.js`

**What's wrong:** React Compiler (`babel-plugin-react-compiler`) is now the officially recommended way to handle memoisation in Expo SDK 54+ apps. It automatically eliminates the need for manual `useMemo`, `useCallback`, and `React.memo`. Not having it means all the memoisation issues in this codebase have to be fixed manually one by one.

**Fix:**
```bash
npx expo install babel-plugin-react-compiler
```

```js
// babel.config.js
module.exports = function (api) {
  api.cache(true)
  return {
    presets: ['babel-preset-expo'],
    plugins: [
      ['babel-plugin-react-compiler', { target: '19' }],
      'react-native-reanimated/plugin',
    ],
  }
}
```

Note: React Compiler will automatically skip components it can't safely optimise. Run `npx react-compiler-healthcheck` first to see if any components break the rules of hooks.

#### M1 — Untracked `setTimeout` calls in `HomeScreenBackground` (memory leak)

**File:** `src/screens/HomeScreen/components/HomeScreenBackground/index.tsx:103,125`

**What's wrong:** Two `setTimeout` calls inside `useCallback` handlers have no cleanup. If the component unmounts while the timer is pending, the callback fires and calls `setCelebrationTrigger(false)` on an unmounted component.

```tsx
// Line 103 — no cleanup
setTimeout(() => {
  setCelebrationTrigger(false)
  hasTriggeredRef.current = false
}, 1600)
```

**Fix:**
```tsx
const celebrationTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null)

// In useEffect cleanup:
useEffect(() => {
  return () => {
    if (celebrationTimerRef.current) clearTimeout(celebrationTimerRef.current)
  }
}, [])

// Replace both setTimeout calls:
celebrationTimerRef.current = setTimeout(() => {
  setCelebrationTrigger(false)
  hasTriggeredRef.current = false
}, 1600)
```

#### M7 — Production errors fully suppressed with no crash reporting

**File:** `app/_layout.tsx:19–25`

**What's wrong:**
```tsx
if (!__DEV__) {
  console.log = () => {}
  console.warn = () => {}
  console.error = () => {}  // ← crashes are silent in production
  // ...
}
```

Firebase Cloud Function errors, network failures, and unexpected exceptions all vanish in production builds. There's no way to know something went wrong for a real user.

**Fix:** Integrate Sentry (Expo-native) before silencing logs:
```tsx
import * as Sentry from '@sentry/react-native'

if (!__DEV__) {
  Sentry.init({ dsn: process.env.EXPO_PUBLIC_SENTRY_DSN })
  // Keep console.error pointing to Sentry
  const originalError = console.error
  console.error = (...args) => {
    Sentry.captureException(args[0])
    // Optionally suppress from console still:
    // originalError(...args)
  }
}
```
